v0.4.0 — Full SymEngine on Android + Windows

The release where the help-mode popovers stop lying. Pre-v0.4.0,
tapping any CAS button in help mode showed "Computed via
SymEngine.X" regardless of whether SymEngine was actually
reachable on the user's platform — and on Android, Windows, and
Linux it wasn't. v0.4.0 makes that promise real on Android arm64
and Windows x86_64.

Headline changes since v0.3.0:

**SymEngine bridge — PLAN P11**

- **R132 — Android arm64-v8a SHIPPED.** Bridge plugin now ships
  `libsymbolic_math_bridge.so` (17 MB stripped) at
  `android/src/main/jniLibs/arm64-v8a/`. CI workflow at
  `.github/workflows/build-android.yml` (`ubuntu-latest` + vcpkg
  manifest install of `symengine[flint,mpfr]` against the
  `arm64-android-release` triplet + Android NDK chainload).
  ~14 min cold-cache CI build. 7 iterations to green; full debug
  log in the bridge plugin's `ANDROID_STATUS.md`.
- **R131 — Windows x86_64 SHIPPED.** Bridge plugin now ships
  `symbolic_math_bridge_plugin.dll` (5.7 MB stripped PE32+) at
  `windows/Libraries/`. CI workflow at
  `.github/workflows/build-windows.yml` (`windows-latest` +
  MSYS2/MinGW64 + pacman pre-built flint/mpfr/gmp/mpc/boost +
  source-built SymEngine). ~7 min cold-cache CI build. (The
  vcpkg+MSVC path was abandoned after 6 attempts kept hitting
  GHA's 6-hour Windows runner cap.) Plain C ABI loadable from
  MSVC-built Flutter Windows apps via dart:ffi.
- **Bridge pin bumped**: `pubspec.yaml` ref `505074d` → `85bfa7e`
  (bridge v1.1.0).

**What's still degraded**

- **Linux x86_64** — PLAN P11 R130 picks up the same vcpkg path
  Android took, without the NDK chainload (host IS Linux). 1
  day of work; deferred.
- **Android x86_64 / armeabi-v7a** — extend the bridge's build
  matrix when emulator or 32-bit devices need testing. Currently
  only arm64-v8a (covers ~99% of modern Android phones).

**Test count**

1992 unit tests pass (unchanged from v0.3.0 — the bridge pin
bump doesn't affect tests since the bridge is unavailable in the
test VM regardless of pin).

**Smoke-tested vs not**

Compile + link + symbols-in-export-table verified in CI for both
new platforms (`llvm-nm` on Android, `objdump -p` on Windows;
all 40+ `flutter_symengine_*` symbols present). **End-to-end
runtime FFI call from a real Android device or Windows desktop
is the natural followup** — not yet validated in this release.
If a runtime issue surfaces, the iteration is to add missing
symbols to `force_link.c` + re-run the bridge workflow.
